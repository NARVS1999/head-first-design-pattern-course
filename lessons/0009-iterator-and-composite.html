Iterator & Composite

Isang Traversal, Iisang Tree — Well-Managed Collections

Lesson Goal Matutunan ang Iterator (traverse collections without exposing structure) at Composite (treat individual objects and groups as one).

Core Concepts

PatternPurposeKey Methods
IteratorMag-traverse ng collection nang hindi ini-expose ang underlying structure.hasNext(), next()
CompositeTreat ng individual objects at groups bilang iisang entity (tree structure).add(), remove(), getChild()

Design Principle

"A class should have only one reason to change."

Sa Taglish: Dapat iisa lang ang dahilan para baguhin ang isang klase — isang responsibilidad lang. Ito ang Single Responsibility Principle.

Fig 1. Composite Pattern — Menu contains MenuItems and sub-Menus recursively

Sample Code 

interface MenuComponent {
    void print();
}

class MenuItem implements MenuComponent {
    String name;
    double price;
    void print() { System.out.println(name + " $" + price); }
}

class Menu implements MenuComponent {
    List<MenuComponent> components = new ArrayList<>();
    void add(MenuComponent c) { components.add(c); }
    void print() {
        for (MenuComponent c : components) c.print(); // recursive!
    }
}

Summary

  • Iterator: Traverse collections nang hindi ini-expose ang structure — hasNext(), next().
  • Composite: Treat individual objects at groups bilang iisang entity — recursive tree structure.
  • Single Responsibility Principle: Isang klase, isang dahilan para baguhin.
